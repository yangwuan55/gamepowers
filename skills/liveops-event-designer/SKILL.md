# LiveOps 活动设计（严格流程）

## 适用条件
- 需要规划赛季节奏、活动矩阵、奖励策略与复盘机制。
- 需要连接活动目标与玩法、经济、内容投放。

## 输入契约
必填输入：
- 经济模型、内容管线、核心循环文档。
- 运营目标（留存、回流、付费、活跃）及优先级。
- 活动上线约束（频率、窗口、资源预算）。

可选输入：
- 历史活动复盘、玩家分层行为数据。

输入缺失处理：一次只问一个问题，不并发追问。

## 设计步骤
1. 活动类型矩阵：定义常驻、限时、赛季、联动活动类型。
完成定义：每类活动有明确目标与触发策略。
2. 赛季节奏设计：规划赛季阶段、峰值窗口和过渡策略。
完成定义：赛季节奏与版本发布节奏可对齐。
3. 奖励与门槛设计：定义活动奖励、参与门槛与回收机制。
完成定义：奖励不破坏经济主循环。
4. 反馈闭环设计：定义实时监控指标与复盘机制。
完成定义：活动结束后可产出可执行改进项。
5. 执行任务映射：将活动链路映射为可交付任务。
完成定义：任务具备文档引用、依赖与验收标准。

## 输出工件
- 主文档：`docs/gamepowers/liveops/season-event-framework.md`
- 审查输入：`docs/gamepowers/economy/economy-model.md`（建议区）
- 任务映射证据：`docs/gamepowers/index/task-doc-map.yaml`

## 质量门禁
- 活动矩阵、赛季节奏、奖励策略、复盘机制齐备。
- 活动收益与经济约束一致，无结构性冲突。
- 活动任务可同步到 `bd` 并可追溯。

## 风险与缓解
- 风险：活动奖励挤压常驻系统价值。
缓解：活动奖励绑定回收机制并设总量上限。
- 风险：活动频率过高导致内容与运营过载。
缓解：设定并执行活动并发上限与降级策略。

## 与 ECS/协调层联动
- 运行时活动逻辑任务统一标记 `architecture: ECS` 并关联事件流文档。
- 由 `game-design-orchestrator` 统一排期、依赖管理和 `bd` 同步。
- 与 `ecs-architecture-designer` 对齐活动状态机和调度负载。

## 反模式（禁止）
- 只定义活动创意，不定义执行门槛和复盘闭环。
- 覆盖经济模型主结论。
- 活动任务未映射就进入实现。

## 完成判定
- 活动体系可持续迭代且可量化评估。
- 与经济、内容、玩法约束保持一致。
- 映射与 ECS 约束完整。
