# 游戏愿景对齐（严格流程）

## 适用条件
- 项目处于立项或重构阶段，需要明确“做什么、不做什么”。
- 多个领域（玩法、角色、经济、运营）出现方向分歧。

## 输入契约
必填输入：
- 游戏品类、目标平台、目标市场。
- 商业目标（留存/付费/DAU 等）与成功指标优先级。
- 现有约束（工期、团队规模、技术约束）。

可选输入：
- 竞品拆解、用户访谈、历史版本数据。

输入缺失处理：一次只问一个问题，不并发追问。

## 设计步骤
1. 愿景目标定义：统一项目北极星指标与体验目标。
完成定义：形成“目标指标 + 体验陈述 + 非目标列表”。
2. 目标玩家分层：定义核心人群、次级人群与拒绝人群。
完成定义：每类人群都有行为特征与需求动机。
3. 核心幻想与差异化：提炼 1-2 条核心幻想与至少 3 个差异化卖点。
完成定义：每个卖点可映射到可实现玩法或内容。
4. 体验边界收敛：定义强约束（必须满足）与禁区（明确不做）。
完成定义：形成跨领域可执行边界清单。
5. 可追溯落地：把愿景条目映射到后续文档和任务。
完成定义：每条愿景至少关联一个文档路径或任务键。

## 输出工件
- 主文档：`docs/gamepowers/gameplay/gameplay-vision.md`
- 协同索引更新：`docs/gamepowers/index/domain-index.md`
- 任务映射证据：`docs/gamepowers/index/task-doc-map.yaml` 中对应条目

## 质量门禁
- 愿景目标、玩家分层、差异化与边界四部分齐全。
- 所有核心结论可追溯到文档路径，禁止口头约定。
- 与 `task-doc-map.yaml` 的映射关系完整，且可同步到 `bd`。

## 风险与缓解
- 风险：愿景过泛导致后续文档不可执行。
缓解：要求每条愿景绑定可验证指标或玩法落点。
- 风险：不同领域对目标玩家理解不一致。
缓解：统一采用同一玩家分层定义并在索引中固化。

## 与 ECS/协调层联动
- 向 `game-design-orchestrator` 回传愿景边界，作为领域路由输入。
- 对技术相关愿景条目标记 `architecture: ECS`，供 `ecs-architecture-designer` 审查。
- 更新后的任务必须通过 `task-doc-map.yaml -> bd` 同步链路。

## 反模式（禁止）
- 只给口号，不给可验证目标。
- 直接覆盖 `core-loop` 或 `mechanics` 的主结论。
- 未完成映射就进入实现交接。

## 完成判定
- 愿景文档可被其他领域直接引用。
- 关键愿景条目全部可追溯到文档/任务。
- 与协调层和架构层的联动信息完整。
