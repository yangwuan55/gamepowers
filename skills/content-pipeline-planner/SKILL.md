# 内容管线规划（严格流程）

## 适用条件
- 需要建立关卡、敌人、掉落、配置等内容生产流程。
- 需要平衡产能、质量与版本投放节奏。

## 输入契约
必填输入：
- 循环与成长文档。
- 版本节奏目标与上线窗口。
- 团队产能边界（人力、工具、验证成本）。

可选输入：
- 历史产能数据、返工率、缺陷统计。

输入缺失处理：一次只问一个问题，不并发追问。

## 设计步骤
1. 内容单位标准化：定义关卡/敌人/奖励等内容模板。
完成定义：每类内容都有统一字段和验收模板。
2. 生产流水线设计：定义策划、实现、验证、发布的阶段流。
完成定义：每阶段输入/输出和责任边界清晰。
3. 产能模型建立：估算单位内容成本与版本总产能。
完成定义：有容量上限与超载预警规则。
4. 质量检查点定义：定义关键质检门禁与返工流程。
完成定义：问题可追踪到阶段节点和责任归口。
5. 投放节奏收敛：把内容批次映射到任务和版本计划。
完成定义：每个批次都有任务依赖和上线条件。

## 输出工件
- 主文档：`docs/gamepowers/levels/level-design-framework.md`
- 审查输入：`docs/gamepowers/liveops/season-event-framework.md`（建议区）
- 任务映射证据：`docs/gamepowers/index/task-doc-map.yaml`

## 质量门禁
- 模板标准、流水线、产能模型、质检门禁齐备。
- 产能估算有依据，且与版本节奏一致。
- 任务映射可同步到 `bd` 并可追溯。

## 风险与缓解
- 风险：内容生产节奏失控导致上线延期。
缓解：按批次设置硬门禁与降级发布方案。
- 风险：质检后置导致返工堆积。
缓解：将关键质检前置到模板与中期评审节点。

## 与 ECS/协调层联动
- 涉及运行时生成或动态加载的任务标记 `architecture: ECS`。
- 通过 `game-design-orchestrator` 同步批次计划到 `task-doc-map.yaml` 与 `bd`。
- 与 `ecs-architecture-designer` 对齐内容加载、系统调度与性能预算。

## 反模式（禁止）
- 只给排期，不定义内容模板和质检门禁。
- 直接覆盖 LiveOps 主文档结论。
- 产能超载仍强行维持原投放节奏。

## 完成判定
- 内容管线可持续支撑版本生产。
- 产能与质量门禁可执行且可审计。
- 任务映射与 ECS 约束完整。
